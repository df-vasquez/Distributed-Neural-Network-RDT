PROYECTO: Protocolo RDT-UDP para Entrenamiento Distribuido de Redes Neuronales

--------------------------------------------------------------------

1. OBJETIVO

El objetivo de este documento es definir un protocolo de transferencia confiable
(RDT-UDP) implementado en la capa de aplicación sobre UDP.

Este protocolo permite construir un canal confiable sobre un medio no confiable,
incorporando mecanismos de:

- Control de errores
- Control de orden
- Confirmación de recepción
- Retransmisión de paquetes perdidos
- Detección de corrupción de datos

El diseño está orientado a su implementación directa en C++ usando sockets.

--------------------------------------------------------------------

2. MODELO DE RED

La arquitectura del sistema corresponde a un modelo tipo Parameter Server:

- 1 Nodo Maestro (servidor central)
- 3 Nodos Esclavos (workers distribuidos)

El sistema opera de forma síncrona, donde el maestro coordina el flujo completo
del entrenamiento distribuido.

--------------------------------------------------------------------

3. UNIDAD DE TRANSMISIÓN (DATAGRAMA)

Cada mensaje del protocolo tiene un tamaño fijo obligatorio:

500 bytes exactos

Estructura del datagrama:

Bytes 0 - 1    → FLAGS (2 bytes)
Bytes 2 - 5    → SEQ (4 bytes)
Bytes 6 - 9    → ACK/NACK (4 bytes)
Bytes 10 - 498 → PAYLOAD (489 bytes)
Byte 499       → CHECKSUM (1 byte)

--------------------------------------------------------------------

4. DESCRIPCIÓN DE CAMPOS

4.1 FLAGS (2 bytes)

Define el tipo de paquete:

- "01" → inicio de transmisión
- "00" → paquete intermedio
- "11" → fin de transmisión
- "10" → paquete de control (ACK/NACK)

Permite agrupar múltiples paquetes en una misma transmisión lógica.

--------------------------------------------------------------------

4.2 SEQ (4 bytes)

Número de secuencia del paquete en formato ASCII (0000–9999).

Funciones:
- Ordenar paquetes recibidos fuera de secuencia
- Detectar pérdidas de paquetes
- Permitir reensamblaje correcto del mensaje

--------------------------------------------------------------------

4.3 ACK / NACK (4 bytes)

Campo de control de confirmación:

- ACK: indica el siguiente paquete esperado
- NACK: indica el paquete que debe retransmitirse

Permite recuperación rápida sin depender del timeout.

--------------------------------------------------------------------

4.4 PAYLOAD (489 bytes)

Contiene la información principal del sistema.

Tipos de datos:

- DATASET: fragmentos del conjunto de entrenamiento
- WEIGHTS: parámetros del modelo neuronal
- GRADIENTS: resultados del backward pass

Si el contenido es menor a 489 bytes, se rellena con el carácter '#'.

--------------------------------------------------------------------

4.5 CHECKSUM (1 byte)

Mecanismo de detección de errores.

Se calcula como:

Checksum = (suma de bytes del datagrama [0..498]) mod 256

Uso:
- Detectar corrupción de datos
- Validar integridad del paquete
- Descartar tramas inválidas

--------------------------------------------------------------------

5. MECANISMOS DE CONFIABILIDAD

5.1 Detección de errores
Se utiliza CHECKSUM para validar la integridad de cada paquete recibido.

Si el checksum no coincide, el paquete es descartado.

--------------------------------------------------------------------

5.2 Control de orden
El campo SEQ permite reconstruir el orden correcto de los paquetes
en caso de llegada desordenada (propio de UDP).

--------------------------------------------------------------------

5.3 ACK (Confirmación positiva)
El receptor envía un ACK indicando el siguiente paquete esperado.

Esto permite al emisor avanzar la ventana de envío.

--------------------------------------------------------------------

5.4 NACK (Confirmación negativa)
Si se detecta pérdida o corrupción, el receptor solicita retransmisión
del paquete específico mediante su número de secuencia.

--------------------------------------------------------------------

5.5 TIMEOUT (select)

El emisor utiliza la función select() para implementar temporización no bloqueante.

Si no recibe respuesta en el tiempo definido:
- Se considera pérdida
- Se retransmite el último paquete enviado

--------------------------------------------------------------------

5.6 ALGORITMO DE KARN

El RTT (Round Trip Time) se calcula únicamente con transmisiones exitosas
sin retransmisión.

Esto evita mediciones incorrectas causadas por paquetes duplicados o retrasados.

--------------------------------------------------------------------

5.7 BACKOFF EXPONENCIAL

Si ocurren fallos consecutivos:

Timeout = Timeout * 2

Esto reduce la congestión y estabiliza la transmisión en la red local.

--------------------------------------------------------------------

6. FLUJO DEL PROTOCOLO EN EL SISTEMA

1. El maestro divide el dataset en partes iguales.
2. Envía cada parte a los nodos esclavos mediante RDT-UDP.
3. Envía los pesos iniciales del modelo.
4. Cada esclavo ejecuta entrenamiento local.
5. Los esclavos envían gradientes al maestro.
6. El maestro valida, reordena y promedia gradientes.
7. Se actualiza el modelo global.

--------------------------------------------------------------------

7. CONSIDERACIONES DE IMPLEMENTACIÓN

- Todos los paquetes deben tener exactamente 500 bytes.
- El orden de bytes es estrictamente obligatorio.
- El checksum se valida antes de procesar el payload.
- El sistema debe ser tolerante a pérdida y desorden de paquetes.
